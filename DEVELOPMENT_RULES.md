# Animify 长期开发规则

> 最后更新：2026-07-24  
> 适用范围：Animify 的 Work 规划、Codex 本地开发、用户测试与 GitHub 维护

本文档是 Animify 长期开发规则。

任何参与 Animify 开发的 Work 或 Codex 在开始工作前，必须完整阅读：

1. `DEVELOPMENT_RULES.md`
2. `PROJECT_STATUS.md`
3. 当前本地 Git 状态
4. 当前任务相关的真实代码

如果本文档与临时开发建议冲突，以用户在当前任务中的明确要求为最高优先级。

---

## 一、三方职责

### 1. 用户职责

用户主要负责：

- 描述产品想法。
- 描述实际看到的现象。
- 提供截图、录屏、浏览器日志或操作步骤。
- 按照测试要求在真实环境中测试。
- 反馈实际测试结果。
- 决定产品体验是否符合预期。
- 对 Git commit、push、PR、merge 等操作给予明确授权。
- 告知是否存在尚未提交或尚未 push 的本地修改。
- 在需要时决定是否继续、暂缓或调整需求。

用户不需要：

- 自己定位 Bug 根因。
- 自己决定代码应该改在哪个文件。
- 自己设计内部数据结构。
- 自己判断问题属于前端状态、动画编译、持久化还是导出。
- 在 Codex 可以直接修改项目时手动复制大量代码。

### 2. Work 职责

Work 主要负责产品和开发协调：

- 理解 Animify 的长期目标。
- 维护开发路线。
- 判断用户反馈属于：
  - 正常行为；
  - 已知限制；
  - UX 缺陷；
  - 功能 Bug；
  - 数据一致性问题；
  - 性能问题；
  - 应立即开发；
  - 应暂缓开发。
- 选择实现方案。
- 明确功能边界。
- 排定开发顺序和依赖。
- 防止需求无序扩张。
- 给 Codex 提供准确开发任务。
- 根据 Codex 的实现报告和用户测试结果更新 `PROJECT_STATUS.md`。
- 保证计划、代码状态和用户测试状态不混淆。
- 在没有读取真实代码时，不猜测文件、函数、字段或实现细节。
- 不得把“计划开发”描述成“正在开发”。
- 不得把“代码已实现”描述成“用户已验证”。

Work 不应默认自己能够读取用户电脑中的本地文件。

只有当前环境明确接入项目文件或 GitHub 后，Work 才能声明已经读取相应内容。

### 3. Codex 职责

Codex 主要负责本地代码执行：

- 开始前读取 `PROJECT_STATUS.md` 和 `DEVELOPMENT_RULES.md`。
- 检查真实 Git 状态。
- 读取任务相关代码。
- 提取准确文件路径、函数名、类型名和数据结构。
- 根据现有架构选择实现方式。
- 编写和修改代码。
- 添加必要且有价值的注释。
- 保持代码可读、可维护、命名明确。
- 遵守项目现有 TypeScript、ESLint、Prettier 和文件组织规范。
- 优先修复根因。
- 避免临时补丁掩盖状态错误。
- 运行 Build、Lint 和相关测试。
- 检查实际 diff。
- 保护用户未提交修改。
- 完成一个可测试阶段后停止。
- 提供准确测试步骤。
- 报告已知限制和未验证内容。
- 根据实际代码状态更新 `PROJECT_STATUS.md`。
- 未经允许不得执行 Git 写入或发布操作。

Codex 不负责替用户宣布测试通过。

## 二、Work 与 Codex 的上下文衔接

Work 对话和 Codex 本地开发环境不能假定会自动共享完整上下文。

以下两个文件是长期衔接机制：

```text
PROJECT_STATUS.md
DEVELOPMENT_RULES.md
```

### `PROJECT_STATUS.md` 负责记录

- 当前 GitHub 基线
- 当前本地基线
- 已完成功能
- 用户验证状态
- 正在开发的功能
- 下一步任务
- 依赖关系
- 待优化项目
- 暂缓需求
- 已知问题
- 最近测试结果
- 下一位开发者入口

### `DEVELOPMENT_RULES.md` 负责记录

- 长期协作规则
- Git 安全规则
- 代码质量要求
- 测试与验收规则
- 状态更新规则
- Work 与 Codex 的职责边界

### 上下文循环

标准循环为：

1. Work 根据用户想法和现象确定任务。
2. Work 更新或生成明确的开发说明。
3. Codex 读取两个维护文件和真实代码。
4. Codex 修改本地代码。
5. Codex 完成代码级验证。
6. Codex 停止并提供测试步骤。
7. 用户在实际浏览器中测试。
8. 用户把结果反馈给 Work 或 Codex。
9. Work 判断是正常行为、Bug、UX 缺陷还是暂缓需求。
10. Codex根据判断继续修复或进入下一阶段。
11. `PROJECT_STATUS.md` 同步更新。
12. 用户明确允许后再进行 Git 操作。

任何重要决定如果只存在于聊天中、没有进入维护文件，就可能在切换环境时丢失。因此阶段性结论必须写入维护文件。

## 三、开发状态必须明确区分

所有任务必须使用以下状态之一。

### 1. 计划开发

含义：

- 已经进入路线图。
- 尚未开始修改代码。

允许表述：

```text
计划开发
下一步计划
尚未开始
```

禁止表述：

```text
正在完成
基本完成
已实现
```

### 2. 正在开发

含义：

- 已经读取代码并开始产生实际修改。
- 尚未形成完整测试阶段。

必须记录：

- 正在修改的功能
- 涉及文件
- 当前阻塞
- 下一步动作

### 3. 代码实现完成，待用户验证

含义：

- 代码已经实现。
- Build、Lint 和相应自动化测试已经通过。
- 尚未取得用户真实操作测试结果。

必须明确写：

```text
代码实现完成，待用户验证
```

不得缩写成“功能完成”。

### 4. 用户验证发现问题

含义：

- 用户实际测试后发现异常或体验不符合预期。

必须记录：

- 操作步骤
- 实际现象
- 预期现象
- 是否稳定复现
- 浏览器控制台信息
- 初步分类
- 后续处理方案

### 5. 已验证完成

只有满足以下全部条件才能使用：

- 代码已经实现；
- Build 通过；
- Lint 通过；
- 相关自动化测试通过，或明确记录项目暂时没有该测试；
- 用户按照测试要求实际验证；
- 用户明确表示结果符合预期；
- 已知限制已经记录；
- `PROJECT_STATUS.md` 已更新。

### 6. 已 push

“已验证完成”和“已 push”是两个不同状态。

可能出现：

```text
代码实现完成，待用户验证，但尚未 commit
用户验证完成，但尚未 push
已经 commit，但尚未 push
已经 push，但仍需扩大回归范围
```

不得混淆。

## 四、开发节奏

### 1. 一次只开发一个可测试阶段

一个阶段应满足：

- 目标明确；
- 边界明确；
- 能独立测试；
- 不依赖大量未完成能力；
- 出现问题时容易定位；
- 可以独立撤回或调整。

不得同时铺开多个无法分别验证的大功能。

### 2. 标准开发流程

每一个阶段必须按以下顺序执行：

1. 阅读两个维护文件。
2. 检查 Git 状态。
3. 读取相关代码。
4. 确认准确的数据流和调用关系。
5. 判断问题类型。
6. 制定最小完整实现方案。
7. 明确不会修改的范围。
8. 修改代码。
9. 添加必要注释。
10. 检查 TypeScript 类型。
11. 运行格式检查。
12. 运行 ESLint。
13. 运行 Build。
14. 运行相关自动化测试。
15. 检查最终 diff。
16. 更新 `PROJECT_STATUS.md`。
17. 停止开发。
18. 给用户明确测试步骤。
19. 等待用户实际反馈。
20. 根据反馈继续或结束。

### 3. 完成可测试阶段后必须停止

出现以下情况之一时必须停止：

- 一个独立功能已经达到可测试状态；
- 需要用户判断交互是否符合预期；
- 需要用户提供浏览器现象；
- 需要用户提供控制台日志；
- 即将执行 commit；
- 即将执行 push；
- 即将创建 PR；
- 即将 merge；
- 需要进行可能影响用户修改的 Git 操作；
- 发现需求范围与原计划明显不同。

停止时必须提供：

- 本阶段实现内容；
- 修改文件列表；
- 代码检查结果；
- 自动化测试结果；
- 明确的手动测试步骤；
- 预期结果；
- 需要特别观察的边界；
- 当前仍未验证的内容；
- 下一步建议。

## 五、状态维护机制

### 1. 每次开始开发前

Codex 必须确认：

- `PROJECT_STATUS.md` 是否存在；
- `DEVELOPMENT_RULES.md` 是否存在；
- 当前任务是否与状态文件一致；
- 当前 Git 分支；
- 当前 HEAD；
- `origin/main` 状态；
- 工作区是否有修改；
- 是否有未跟踪文件；
- 是否有暂存文件；
- 是否有本地未 push 提交。

### 2. 每次代码实现后

`PROJECT_STATUS.md` 至少更新：

- 当前功能状态；
- 修改文件；
- 测试命令与结果；
- 用户测试是否完成；
- 已知问题；
- 下一步入口；
- 当前 Git 状态；
- commit 和 push 状态。

### 3. 用户测试后

如果用户测试通过：

- 把对应功能更新为“已验证完成”。
- 记录测试日期。
- 记录关键测试范围。
- 记录仍然存在的限制。

如果用户测试失败：

- 不得保留“已验证完成”。
- 改为“用户验证发现问题”。
- 记录实际现象。
- 记录下一步诊断入口。

### 4. 每次结束工作时

必须在 `PROJECT_STATUS.md` 中明确列出：

- 已完成
- 正在开发
- 待开发
- 暂缓
- 最近测试
- Git 状态
- 下次第一步

不得只在聊天中口头总结。

### 5. 修改规则文件

`DEVELOPMENT_RULES.md` 只在以下情况修改：

- 用户明确增加或修改长期规则；
- 原规则存在冲突；
- 新协作环境需要明确职责；
- Git 安全规则需要加强；
- 发现现有流程持续导致状态丢失。

不得因为一次临时任务随意改动长期规则。

## 六、Git 安全规则

### 1. 未经用户明确允许，禁止执行

以下操作必须获得用户对当前具体操作的明确授权：

```text
git add
git commit
git push
创建 Pull Request
更新 Pull Request
合并 Pull Request
git merge
git rebase
git cherry-pick
git reset
git clean
git restore
git checkout -- <file>
删除分支
强制推送
改写提交历史
删除或覆盖标签
```

用户要求“修改代码”不等于允许 commit。

用户要求“继续开发”不等于允许 push。

用户允许 commit 不等于允许 push。

用户允许 push 不等于允许创建 PR 或 merge。

### 2. 明确授权用语

只有类似以下明确表达才构成授权：

```text
可以 commit
可以 push
可以创建 PR
可以 merge
可以删除这个分支
可以执行你刚才说明的 reset 操作
```

授权只对当前已说明的目标和范围有效。

不得把以前某次授权自动沿用到后续任务。

### 3. 允许的只读检查

通常可以执行：

```text
git status
git diff
git diff --cached
git log
git show
git branch --show-current
git remote -v
```

`git fetch` 虽然不会修改工作树，但会更新远端引用。执行前应说明用途，尤其是在存在本地分叉时。

### 4. 保护用户修改

如果工作区存在用户修改：

- 默认这些修改属于用户。
- 不得覆盖。
- 不得删除。
- 不得格式化整个项目导致无关 diff。
- 不得把不相关修改混入当前任务。
- 不得擅自暂存。
- 不得擅自提交。
- 不得使用 reset 或 clean 清除。
- 修改同一文件前必须先查看 diff。
- 如果无法安全避开，必须停止并询问用户。

### 5. 禁止危险路径操作

不得对以下范围执行递归删除或覆盖：

- 用户主目录
- 工作区根目录
- Git 仓库根目录
- 未解析的环境变量路径
- 不明确的通配符路径

任何删除操作必须先确认准确目标和恢复方式。

### 6. 分支分叉处理

当本地与远端同时存在独立提交时：

- 先读取日志和 diff。
- 明确哪些提交属于用户。
- 明确哪些提交已经在远端被等效实现。
- 必要时先创建可恢复备份。
- 不得直接 pull、rebase 或 reset。
- 向用户说明可选方案和影响。
- 获得许可后再执行。

### 7. Push 前检查

用户允许 push 后，仍必须检查：

```text
当前分支
当前 HEAD
待 push 的提交列表
工作区是否干净
Build 是否通过
Lint 是否通过
测试是否通过
远端是否出现新提交
```

如果远端在用户测试后发生变化，应停止并重新评估。

## 七、代码质量规则

### 1. 必须添加必要且有价值的注释

注释应该解释：

- 为什么这样设计；
- 状态由谁负责；
- 数据的权威来源；
- 生命周期边界；
- 兼容旧数据的原因；
- 浏览器 API 的特殊行为；
- 容易被误改的约束；
- 为什么必须清理资源或动画实例；
- 为什么某个操作不进入 Undo；
- 为什么某个边界值必须被限制；
- 预览、放映和导出之间如何保持一致。

不应添加：

- 重复代码字面含义的注释；
- 与真实行为不一致的历史注释；
- 没有信息量的注释；
- 为每一行代码机械添加的注释；
- 无法验证的推测性注释。

### 2. 可读性

代码必须：

- 使用明确命名；
- 避免含糊缩写；
- 避免超长函数；
- 避免深层嵌套；
- 让状态所有权清晰；
- 让命令入口清晰；
- 让错误边界清晰；
- 让清理逻辑清晰；
- 让数据转换方向清晰。

### 3. 可维护性

优先：

- 复用现有类型；
- 复用现有命令层；
- 复用现有编译器；
- 复用统一播放控制器；
- 提取真正共享的规则；
- 将纯计算与 React UI 分离；
- 将项目数据和临时预览状态分离；
- 将用户持久化状态和运行时状态分离；
- 保持画布、放映和导出行为一致。

禁止：

- 为同一规则创建多套实现；
- 在 UI 中绕过命令层直接修改复杂动画数据；
- 用临时变量掩盖数据同步问题；
- 用刷新 key 长期替代正确状态清理；
- 复制整段逻辑到编辑器和导出端；
- 为快速通过测试写不可维护补丁。

### 4. TypeScript

必须：

- 使用项目已有类型；
- 保持联合类型穷尽处理；
- 处理旧项目可选字段；
- 对外部数据进行边界检查；
- 避免无意义类型断言；
- 避免扩大到 `any`；
- 明确函数输入和返回值；
- 对运行时可能缺失的数据进行保护。

只有存在明确理由并记录说明时，才允许局部使用宽泛类型。

### 5. ESLint 和 Prettier

必须遵守项目现有配置：

- 不得关闭规则来掩盖问题。
- 不得为了当前任务修改大量 ESLint 规则。
- 不得未经讨论引入新的全局格式规则。
- 如果存在 Prettier 配置，使用现有配置。
- 如果没有 Prettier 配置，不得假设存在对应命令。
- 格式化只作用于相关文件，避免无关 diff。

### 6. 文件组织

必须遵守现有目录职责：

```text
src/components/editor
src/hooks
src/types
src/utils
src/data
```

一般职责：

- React UI 放在 `components`。
- 可复用状态控制放在 `hooks`。
- 数据类型放在 `types`。
- 纯函数、命令、编译和导出逻辑放在 `utils`。
- 示例数据放在 `data`。

新增文件前必须检查项目是否已有相同职责的文件。

### 7. 明确命名

命名必须表达真实含义。

禁止：

- 猜测现有标识符；
- 随意改变字段大小写；
- 对路径、键名或 JSON 结构进行模糊匹配；
- 使用无法表达职责的名称；
- 使用临时名称长期保留；
- 使用不确定措辞代替真实状态。

如果无法从代码、日志或测试数据中取得准确名称，必须询问用户，不得自行猜测。

## 八、根因修复规则

### 1. 优先修复根因

面对问题时必须先判断：

- 数据源是否错误；
- 状态所有权是否重复；
- 是否存在旧状态未清理；
- UI 是否读取了错误数据；
- 命令层是否没有同步更新；
- 编译器是否忽略了合法数据；
- React key 是否掩盖状态问题；
- 本地持久化是否恢复了旧数据；
- 导出端是否复制了过期逻辑；
- 浏览器 API 是否保留了旧动画实例。

不得看到 UI 不更新就默认增加强制刷新。

### 2. 临时规避必须明确记录

如果因为阶段限制必须使用临时方案：

- 必须说明原因；
- 必须说明影响范围；
- 必须写入 `PROJECT_STATUS.md`；
- 必须列出后续移除条件；
- 不得把临时方案描述为最终架构。

### 3. 修复后必须验证相邻路径

例如修复动画删除时，应同时检查：

- 属性面板；
- 高级动画工作区；
- Timeline；
- Canvas；
- Undo；
- Redo；
- 保存恢复；
- HTML 导出。

不能只验证用户最先看到的一个位置。

---

## 九、禁止无关重构

### 1. 当前任务范围优先

修改必须服务于：

- 当前功能；
- 当前 Bug 根因；
- 当前功能直接依赖的优化；
- 当前文件中会阻碍正确实现的问题。

### 2. 可以顺带优化的情况

只有满足以下条件时可以插入优化：

- 与当前修改属于同一职责；
- 不处理会明显增加返工；
- 能在同一测试范围内验证；
- 不会引入大面积无关 diff；
- 不会改变未讨论的产品行为。

### 3. 必须暂缓的情况

以下情况应记录到待优化，而不是立即修改：

- 需要改动大量无关文件；
- 需要重新设计整个架构；
- 无法通过当前测试覆盖；
- 会改变用户尚未确认的交互；
- 只是代码风格偏好；
- 没有实际问题证据；
- 会扩大任务到无法独立验收。

---

## 十、问题分类规则

用户只需要报告现象和想法。

Work 或 Codex 必须将其分类为以下之一。

### 1. 正常行为

当前设计明确如此，且没有违反目标。

处理方式：

- 解释当前行为。
- 必要时补充操作方式。
- 不修改代码。

### 2. 阶段性限制

数据结构已经预留，但当前阶段尚未接入。

处理方式：

- 记录限制。
- 判断是否属于当前主线。
- 不把它当作 Bug 修补。

### 3. UX 缺陷

功能能够工作，但操作难理解、反馈不足或容易误操作。

处理方式：

- 明确用户完成目标所需步骤。
- 判断是否可以在当前阶段改进。
- 优先处理高频、低成本且影响明显的问题。

### 4. 功能 Bug

代码行为与已经确定的产品规则不一致。

处理方式：

- 找到稳定复现步骤。
- 定位根因。
- 修复相关数据流。
- 验证相邻路径。

### 5. 数据一致性问题

不同面板、Canvas、Timeline、保存数据或导出结果不一致。

处理方式：

- 确定权威数据源。
- 删除重复状态或错误同步。
- 验证保存、恢复、Undo、Redo 和导出。

### 6. 暂缓需求

功能合理，但不属于当前阶段或依赖尚未完成。

处理方式：

- 写入 `PROJECT_STATUS.md`。
- 说明依赖。
- 不提前开发半套能力。

## 十一、测试规则

### 1. 代码检查不能代替用户测试

以下检查只能证明代码级状态：

- TypeScript 编译通过
- Vite Build 通过
- ESLint 通过
- Prettier 通过
- 单元测试通过
- `git diff --check` 通过

它们不能证明：

- 交互符合用户预期；
- 动画视觉效果正确；
- 浏览器操作没有问题；
- 放映节奏自然；
- 导出文件行为一致。

### 2. 每个阶段必须提供明确测试步骤

测试说明必须包含：

- 测试前准备；
- 点击位置；
- 输入值；
- 操作顺序；
- 预期画面；
- 预期状态；
- Undo 测试；
- Redo 测试；
- 刷新恢复测试；
- 删除与切换测试；
- 导出测试；
- 需要观察的异常。

禁止只说：

```text
请测试一下
看看是否正常
试一下动画
```

### 3. 测试必须可重复

如果用户报告问题，应记录：

```text
测试环境
当前页面
选中的对象
动画类型
Clip 数量
关键帧数量
操作顺序
实际结果
预期结果
是否每次出现
控制台错误
```

### 4. 未经用户测试不得宣布完成

允许表述：

```text
代码实现完成，待用户验证
Build 和 Lint 已通过
自动化测试已通过
```

禁止表述：

```text
功能已经完全完成
问题已经彻底解决
现在肯定不会再出现
```

除非用户已经实际测试并确认。

### 5. 测试失败后的处理

用户测试失败时：

- 先记录现象。
- 不立即猜原因。
- 检查真实代码和日志。
- 判断是正常行为、UX、Bug 还是阶段限制。
- 必要时要求用户提供截图、录屏或控制台日志。
- 修复后重新提供针对性测试步骤。
- 原有“已验证完成”状态必须撤回。

## 十二、手动提供代码时的规则

只有 Codex 无法直接访问项目文件时，才采用手动粘贴方式。

提供修改时必须包含：

1. 准确文件路径。
2. 可直接搜索的真实原始代码。
3. 修改位置前后上下文。
4. 明确说明是插入、替换还是删除。
5. 完整代码块。
6. 需要新增的 import。
7. 需要删除的旧代码。
8. 修改后的测试步骤。

不得只提供行号，因为文件变化后行号可能失效。

不得提供模糊定位，例如：

```text
在合适的位置加入
在相关函数附近修改
找到类似代码
```

如果无法取得真实查找点，必须让用户提供对应最新版文件。

## 十四、当前 Animify 固定协作原则

1. 用户负责反馈现象和想法。
2. Work 负责判断问题性质和开发优先级。
3. Codex 负责读取真实代码、实施和代码级验证。
4. 不猜测标识符、路径、字段和数据结构。
5. 不确定时优先读取真实文件。
6. 无法从文件中确认时询问用户。
7. 代码必须包含必要且有价值的注释。
8. 代码必须可读、可维护、命名明确。
9. 遵守现有 TypeScript、ESLint、Prettier 和文件组织规范。
10. 优先修复根因。
11. 禁止无关重构。
12. 保护用户所有未提交修改。
13. 一次完成一个可测试阶段。
14. 完成阶段后停止并给出明确测试步骤。
15. Build 和 Lint 通过不等于用户验证通过。
16. 未经用户实际测试，不得把代码实现完成写成功能验证完成。
17. 未经允许，不得 commit。
18. 未经允许，不得 push。
19. 未经允许，不得创建或更新 PR。
20. 未经允许，不得 merge。
21. 未经允许，不得 reset、clean、rebase、cherry-pick 或覆盖文件。
22. 每次结束必须更新项目状态。
23. 每次开始必须读取项目状态。
24. 编辑器、放映和 HTML 导出必须尽量共用同一规则。
25. 任何重要架构决定必须进入长期维护文件。
